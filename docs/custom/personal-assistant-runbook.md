## 1. VM IP changed

The VM IP is hardcoded in these places:

| Location | What to update |
|---|---|
| `deploy-all.ps1` line 13 | `$vmIP = "192.168.122.82"` |
| `~/.cursor/mcp.json` on Windows | The SSH `args` host string |
| `vm-deploy/README.md` | VM details table |
| `vm-deploy/sign-off-checklist.md` | SparkyFitness web UI URL |
| `vm-deploy/send-wa-sparky.sh` | Default message URL |

### Steps

```powershell
# 1. Find the new IP (check Proxmox/VirtualBox/VMware UI or ask)
# 2. Test: ssh -i C:\Users\henza\.ssh\id_rsa henzard@NEW_IP "echo OK"
# 3. Update deploy-all.ps1:
#    Change: $vmIP = "192.168.122.82"
#    To:     $vmIP = "NEW.IP.HERE"
# 4. Update ~/.cursor/mcp.json on Windows:
#    Change the host in the SSH args array
# 5. Reconnect MCP: Ctrl+Shift+P → "MCP: Reconnect servers"
```

The gateway itself doesn't care about the IP — it listens on `localhost:18789`. Only clients connecting from outside (Cursor MCP, browser) need the updated IP.

## 2. Rotate a secret token

### SparkyFitness API key

1. Open `http://<VM_IP>:3004` → Settings → API → Revoke old key → Generate new key
2. On the VM:
   ```bash
   echo "new_token_here" > ~/.openclaw/secrets/sparky-token
   chmod 600 ~/.openclaw/secrets/sparky-token
   ```
3. No gateway restart needed — the MCP server reads the file at call time.
4. Test: run `e2e-test.sh` section 2 (SparkyFitness reads).

### Todoist token

1. todoist.com → Settings → Integrations → Developer → Revoke → Copy new token
2. On the VM:
   ```bash
   echo "new_token_here" > ~/.openclaw/secrets/todoist-token
   chmod 600 ~/.openclaw/secrets/todoist-token
   ```
3. No gateway restart needed.

### Habitica credentials

Habitica credentials are in the systemd service environment, not in a file:
```bash
systemctl --user edit openclaw-gateway.service
# Update the Environment= lines for HABITICA_USER_ID and HABITICA_API_KEY
systemctl --user daemon-reload
systemctl --user restart openclaw-gateway
```

### Gateway token

The gateway token is auto-generated by OpenClaw and stored in `~/.openclaw/openclaw.json`. It doesn't expire. If you need to rotate it, regenerate the config:
```bash
openclaw config set gateway.token "$(openssl rand -hex 24)"
systemctl --user restart openclaw-gateway
```
Then update any scripts that read it dynamically (they all use `python3 -c "... d['token'] ..."` so they'll pick up the new value automatically).

## 3. SparkyFitness stopped working

### Check containers

```bash
docker ps   # all 3 should show "Up"
docker compose -f ~/sparky/docker-compose.yml logs --tail=50 sparkyfitness-server
```

### Container keeps restarting

Usually a postgres version conflict or volume issue:
```bash
docker compose -f ~/sparky/docker-compose.yml down
docker volume ls | grep sparky
docker volume rm <volume_id>
rm -rf ~/sparky/postgresql ~/sparky/backup ~/sparky/uploads
docker compose -f ~/sparky/docker-compose.yml up -d
# Verify postgres:16-alpine override is still in place:
cat ~/sparky/docker-compose.override.yml
```

### API returns 401

Token may have been regenerated. Re-copy from Settings → API in the web UI and update `~/.openclaw/secrets/sparky-token`.

### API returns 404

Route is wrong. See [mcp-implementation-guide.md §12](mcp-implementation-guide.md#12-sparkyFitness-tool--complete-reference) for the full correct route table.

### Goals not loading (200 but empty)

You need to set goals in the web UI first. Go to `http://<VM_IP>:3004` → Goals → set calorie and macro targets.

## 4. Add, change, or remove a cron job

### Add a new cron

```bash
# On the VM (PATH must include openclaw)
export PATH="$HOME/.npm-global/bin:$HOME/.local/bin:$PATH"

openclaw cron add \
  --name "my-new-cron" \
  --cron "0 9 * * 1-5" \
  --tz "Africa/Johannesburg" \
  --session isolated \
  --announce \
  --message "Your prompt here. Use the native <tool> tool. Do NOT edit TOOLS.md."
```

**Rules:**
- Always include "Do NOT edit TOOLS.md" in any cron prompt that touches workspace files.
- Use `--session isolated` for all crons to prevent context bleed between runs.
- Use `--announce` to deliver to WhatsApp. Add `--channel whatsapp --to "$OWNER_WA"` only if sending to a different number.
- Name must be unique. Check `openclaw cron list` first.

### Update an existing cron

```bash
# First, find the cron ID
openclaw cron list

# Update (you can change name, cron schedule, message, etc.)
openclaw cron update <id> --message "New prompt here"
openclaw cron update <id> --cron "0 10 * * 1-5"
```

### Remove a cron

```bash
openclaw cron remove <id>
# Or by name (if supported):
openclaw cron list | grep "cron-name"
openclaw cron remove <id-from-above>
```

### Also update `phase7-crons-v2.sh`

If you add a permanent cron, add it to `docs/custom/vm-deploy/phase7-crons-v2.sh` so it's recreated on fresh deployments. The script is idempotent only if the cron names are unique — if a name already exists, `openclaw cron add` will fail. You can wrap new entries in a check:
```bash
if ! openclaw cron list | grep -q "my-new-cron"; then
  openclaw cron add --name "my-new-cron" ...
fi
```

## 5. Update TOOLS.md

TOOLS.md is the agent's identity file. It lives at `~/.openclaw/workspace/TOOLS.md` on the VM. The canonical source is `docs/custom/vm-deploy/TOOLS.md` in the repo.

**Always edit the repo copy first, then deploy.**

### Workflow

```powershell
# 1. Edit in the repo:
#    docs/custom/vm-deploy/TOOLS.md

# 2. Commit:
#    git add docs/custom/vm-deploy/TOOLS.md
#    git commit -m "docs: update agent routing rules for X"
#    git push origin main

# 3. Deploy to VM:
scp -i C:\Users\henza\.ssh\id_rsa docs\custom\vm-deploy\TOOLS.md "henzard@192.168.122.82:~/.openclaw/workspace/TOOLS.md"
```

The gateway picks up the change immediately on the next agent turn — no restart needed.

**Never edit `~/.openclaw/workspace/TOOLS.md` directly on the VM** — it will be overwritten on the next deployment and your changes will be lost.

## 6. Annual calendar refresh

Every January, update the feast dates and any birthday changes for the new year.

### Feast dates

The Seven Feasts follow the Hebrew calendar and shift each year. To find next year's dates:
- Search "Seven Feasts of Israel [YEAR] dates" on a Hebrew calendar converter
- Verify against a reliable source (e.g., hebcal.com — use the Torah/Leviticus dates, not rabbinic additions)
- All dates are sundown-to-sundown. Use the civil calendar date of when sundown begins.

### Update procedure

1. Create `docs/custom/vm-deploy/calendar-YYYY.json` (copy from `calendar-2026.json`)
2. Update feast `start`/`end` dates
3. Update birthday years if tracking ages
4. Update feast cron messages in `phase7-crons-v2.sh` (the `--message` strings include explicit year dates)
5. Remove old feast crons and re-run the feast section of `phase7-crons-v2.sh`:
   ```bash
   # Remove old feast crons by name
   for NAME in passover-7days passover-eve unleavened-bread-7days firstfruits-7days firstfruits-eve \
               shavuot-7days trumpets-7days yom-kippur-7days yom-kippur-morning \
               tabernacles-7days tabernacles-eve; do
     ID=$(openclaw cron list | grep "$NAME" | awk '{print $1}')
     [ -n "$ID" ] && openclaw cron remove "$ID"
   done
   # Re-add with new dates (run just the feast section of phase7-crons-v2.sh)
   ```
6. Deploy the new `calendar-YYYY.json` to the VM:
   ```bash
   scp -i ~/.ssh/id_rsa docs/custom/vm-deploy/calendar-YYYY.json \
     henzard@192.168.122.82:~/.openclaw/workspace/calendar-YYYY.json
   ```

### Birthday crons

Birthday crons use specific day/month in the cron expression (e.g., `0 7 1 8 *` = August 1 every year). These never change unless the person's birthday changes. No annual update needed.

## 7. MEMORY.md and annual goals session

`~/.openclaw/workspace/MEMORY.md` is the agent's dynamic memory. The agent can write to it (it does so during Memory Synthesis crons). You can also write to it directly.

### Structure

```markdown
# Agent Memory

## 2026 Goals
[Populated via guided WhatsApp session]

## Weekly Intentions
Updated each Monday.

## Notes
[Agent writes here during synthesis]
```

### 2026 Goals — guided session

The "Living Your Best Year Ever" (Darren Hardy) framework is built into the agent's weekly briefing crons. To seed the goals:

1. Send this to the WhatsApp agent:
   > "Let's do the annual goals session. Walk me through Living Your Best Year Ever — one life area at a time."

2. The agent will guide you through each area (faith, family, health, finance, career, personal growth). For each area it will ask:
   - Current state (1-10)
   - One big goal for the year
   - One weekly habit that supports it

3. After the session, ask the agent:
   > "Store these goals in MEMORY.md under ## 2026 Goals"

4. Verify: `cat ~/.openclaw/workspace/MEMORY.md`

### Weekly intentions

Every Monday at 6:35am the `weekly-intentions-monday` cron asks for 3 intentions aligned with annual goals. The agent stores them in MEMORY.md under `## Weekly Intentions`. The Sunday `state-of-me-report` cron reviews them.

### Protect MEMORY.md from accidental overwrite

If Memory Synthesis crons are overwriting things they shouldn't, add to the synthesis prompt:
> "Do NOT edit the `## 2026 Goals` section. Only update `## Notes` and `## Weekly Intentions`."

## 8. Deploy a code change to the VM

For TypeScript changes (new tool actions, bug fixes, new extensions):

```powershell
# From Windows repo root:
# 1. Push the code
git push origin main

# 2. Deploy to VM (use the phase2 script)
$sshKey = "C:\Users\henza\.ssh\id_rsa"
$vm = "henzard@192.168.122.82"
scp -i $sshKey docs\custom\vm-deploy\phase2-deploy-code.sh "${vm}:/tmp/oc-phase2.sh"
ssh -i $sshKey $vm "bash /tmp/oc-phase2.sh && rm /tmp/oc-phase2.sh"
```

For MCP server changes only (`tools/openclaw-mcp-server.mjs` — no TypeScript, no build needed):

```powershell
scp -i $sshKey tools\openclaw-mcp-server.mjs "${vm}:~/openclaw-custom/tools/openclaw-mcp-server.mjs"
# Then reconnect in Cursor: Ctrl+Shift+P → "MCP: Reconnect servers"
# No gateway restart needed
```

## 9. Gateway restart and diagnostics

### Restart

```bash
systemctl --user restart openclaw-gateway
```

### Check status

```bash
systemctl --user status openclaw-gateway
curl http://localhost:18789/health
```

### Read logs

```bash
journalctl --user -u openclaw-gateway -n 100 --no-pager
# Follow live:
journalctl --user -u openclaw-gateway -f
```

### Gateway won't start after config change

```bash
openclaw doctor   # shows config validation errors
```

Common cause: "Unrecognized key" error. See [implementation-guide.md §12 B1](implementation-guide.md#b1-config-invalid-unrecognized-key-on-channelswhatsapparchive).

### WhatsApp disconnected

```bash
openclaw channels status   # check session state
# If disconnected: open the OpenClaw web UI or re-scan QR
```

## 11. Contacts change

If Alicia's or Rhyno's number changes, or you add/remove an accountability partner:

1. Update `~/.openclaw/secrets/contacts.env` on the VM:
   ```bash
   nano ~/.openclaw/secrets/contacts.env
   ```

2. Update `TOOLS.md` accountability partners table (in the repo, then SCP to VM).

3. Update the `accountability-audit` cron prompt — it has the accountability partner numbers baked into the `--message` string. Remove and re-add:
   ```bash
   source ~/.openclaw/secrets/contacts.env
   ID=$(openclaw cron list | grep "accountability-audit" | awk '{print $1}')
   openclaw cron remove "$ID"
   # Then re-run just the accountability-audit block from phase7-crons-v2.sh
   ```

4. No code deploy needed — contacts.env is read at script execution time.

## 12. Quick reference — all secret paths

| What | Path on VM | How to get |
|---|---|---|
| Gateway token | `~/.openclaw/openclaw.json` → `token` field | Auto-generated; read with `python3 -c "import json,os; print(json.load(open(os.path.expanduser('~/.openclaw/openclaw.json')))['token'])"` |
| SparkyFitness API key | `~/.openclaw/secrets/sparky-token` | Web UI: Settings → API Keys |
| Todoist API token | `~/.openclaw/secrets/todoist-token` | todoist.com → Settings → Integrations → Developer |
| Phone numbers | `~/.openclaw/secrets/contacts.env` | Your own contacts; see `contacts.env.example` |
| Habitica credentials | systemd service env | Habitica: Settings → API |
| OpenRouter API key | systemd service env | openrouter.ai → Keys |
| WhatsApp session | `~/.openclaw/credentials/whatsapp/` | Scan QR via web UI |

**All secrets live only on the VM.** None are committed to the repo. If you need to set them on a fresh VM, copy the `contacts.env.example`, then add each token as shown above.
